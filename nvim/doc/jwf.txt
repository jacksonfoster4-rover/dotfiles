QUICK REFERENCE — the stuff you actually use                       *jwf-quick*

Leader is  ;  — so ";fn" means press semicolon then f then n.
Stuck? Esc gets you out of almost anything. Ctrl-o jumps back where you were.

  ;ff    Search text in all files (grep) — add rg flags to filter, e.g.

  ;fc    Search for the word under the cursor

  ;tt    Toggle the tree open/closed      ;to  Open (cursor stays in code)
  ;tf    Jump your cursor INTO the tree    ;tc  Close the tree
  ;tn    Reveal current file in the tree   ;fn  fuzzy-find a file (skip the tree)

1. FILE TREE                                                         *jwf-tree*

The file tree (nvim-tree) is a sidebar showing your project directory.
It is separate from the editor — it is its own window, so working with it is
two steps: FOCUS the tree, do something, then jump BACK to your code.

OPENING / FOCUSING / LEAVING:
  ;tt    Toggle tree open/closed
  ;to    Open the tree but KEEP the cursor in your code (just want it visible)
  ;tf    Open if needed AND move the cursor INTO the tree (ready to navigate)
  ;tc    Close the tree
  ;tn    Reveal the current file in the tree
           Use this when you opened a file via search and want to see where
           it lives in the folder structure.

SWITCHING BETWEEN TREE AND CODE (they are side-by-side windows):
  Ctrl-w l    From the tree, jump right into your code
  Ctrl-w h    From your code, jump left into the tree
  Ctrl-w w    Cycle focus through all windows (quick toggle)
  q  (in tree) Close the tree and return to code in one key
           Rule of thumb: ;tf to go in, Ctrl-w l (or open a file) to come back.

MOVING AROUND THE TREE FAST (cursor must be in the tree):
  <count>j/k  Jump by the relative line number shown next to each entry. The
                tree uses relative numbers, so a file 7 rows down shows "7" —
                press 7j to land on it in one go. This is the main fix for
                crawling down one line at a time.
  j / k       Down / up one entry            gg / G   Top / bottom of the tree
  Enter / o   Open file, or expand/collapse the folder under the cursor
  f           Live FILTER — start typing and the tree shrinks to matches.
                F clears the filter. Fastest way to find a known filename.
  S           SEARCH to a node by name and jump straight to it
  /           Vim search within the tree (then n / N for next / prev match)
  P           Jump to the PARENT folder of the current entry
  <  /  >     Jump to previous / next sibling
  J  /  K     Jump to the last / first sibling in the current folder
  E           Expand the whole tree      W   Collapse everything back down
  <C-]>       Make the folder under the cursor the new tree root (cd into it)
  -           Move the tree root UP one directory
  R           Refresh the tree (pick up files changed outside nvim)

FILE OPERATIONS (cursor in the tree):
  a         Create new file (type name; end with / to make a folder instead)
  d         Delete file           r   Rename file
  x         Cut file              c   Copy file          p   Paste file
  y         Copy the filename     Y   Copy the relative path
  I         Toggle hidden files (.dotfiles)    H   Toggle git-ignored files
  g?        Show nvim-tree's own help with every mapping

COMMANDS:
  :NvimTreeFindFile   Scroll the tree to the current buffer's file
  :NvimTreeCollapse   Collapse all open folders

2. SEARCH & FIND                                                    *jwf-search*

All search uses Telescope — a fuzzy finder that shows a live-updating list
as you type. Arrow keys or Ctrl-j/k navigate the results. Enter opens it.

  ;fn    Find file by name (searches filenames recursively from project root)
           Tip: you can type partial paths, e.g. "comp/bu" to match
           "src/components/Button.tsx"

  ;ff    Search for a string inside all files (live grep)
           Supports ripgrep flags — QUOTE your search term when using flags,
           then add the flags after the closing quote.

         INCLUDE ONLY certain file types (-t = ripgrep type):
             "grooming" -t py              only Python files
             "MyFunc" -t ts                only TypeScript files
             "config" -t py -t ts          Python OR TypeScript (repeat -t)
           -t takes a named type, not a glob. List all known type names with
           the shell command:  rg --type-list  (e.g. py, ts, js, html, css).
           To include by pattern instead of a named type, use a glob:
             "grooming" -g *.tsx           only .tsx files

         EXCLUDE files — by type (-T), or by glob/directory (-g !...):
             "grooming" -T json            everything EXCEPT JSON files
             "grooming" -g !*.json         exclude a glob (note the leading !)
             "grooming" -g !some/dir/**    exclude a directory subtree
             "grooming" -g !*.json -g !migrations/**   stack as many as needed

           Without quotes the whole prompt is treated as the literal search
           string and the flags are ignored. node_modules, .git, *.min.js,
           static/dist, venv, and __pycache__ are excluded automatically on
           every search (configured in plugins/telescope.lua) — add more there
           to exclude them permanently instead of per-search.

  ;fc    Search for the word currently under your cursor across all files
           Useful for finding every usage of a variable or function name
           without typing it out.

  ;fs    Fuzzy search inside the current open file only
           Good for jumping to a function or class in a long file.

  ;fo    Symbols in the current file (LSP document symbols)
           A "symbol" is a named thing the language server sees in your code —
           a function, class, method, variable, constant. This lists them for
           the current file so you can jump straight to one by name instead of
           scrolling. VS Code's Ctrl-Shift-O / "Go to Symbol in File". Unlike
           ;fs (which fuzzy-matches raw text), this only lists real definitions.

  ;fw    Symbol definitions across the whole project (ripgrep)
           Type a name and jump to where it's DEFINED anywhere in the project.
           VS Code's Ctrl-T / "Go to Symbol in Workspace". This uses ripgrep,
           not the language server: the LSP workspace-symbol version re-queries
           pyright/ts_ls on every keystroke and is very slow on a big monorepo,
           so ripgrep trades semantic accuracy for near-instant results.
           How it matches: it greps for your name FOLLOWING a definition
           keyword (def, class, function, const, let, type, interface, enum,
           struct…), so it lands on definitions, not every usage. Keep typing to
           narrow. Limitation: JS/TS class methods have no keyword (`foo() {`)
           so they won't show — use ;gd (go to definition) or ;gr for those.

  ;sr    Project-wide search AND replace (nvim-spectre)
           Opens a panel where you can preview every match before replacing.
           Supports regex. Press <leader>R inside spectre to execute the
           replace across all files.

Telescope universal controls:
  Ctrl-c or Esc    Close the picker without opening anything
  Ctrl-u            Scroll the preview pane up
  Ctrl-d            Scroll the preview pane down
  Ctrl-q            Send all results to quickfix list (then :cn/:cp to navigate)
  Tab               Select/deselect items for bulk action (in some pickers)

3. LSP NAVIGATION                                                      *jwf-lsp*

LSP (Language Server Protocol) provides IDE-level features by running a
language-specific server that understands your code. Python runs two: pyright
for types/navigation and ruff for linting + formatting (pyright can't format,
so ruff is what makes format-on-save work on .py). TypeScript/JavaScript use
ts_ls plus eslint. All the commands below only work when a server has attached
to the file you are editing (check :LspInfo).

After any "go to" command, use Ctrl-o to jump back to where you were.
Use Ctrl-i to jump forward again. This works across files.

  ;gd    Go to definition
           Jumps to where a function, class, or variable is defined.
           If there are multiple definitions, a list opens.

  ;gD    Go to declaration
           For most languages this is the same as definition. In C/C++ it
           jumps to the .h header declaration rather than the .c definition.

  ;gi    Go to implementation
           In TypeScript, jumps from an interface method to its concrete
           implementation. In Python, jumps to the actual method body.

  ;gt    Go to type definition
           Jumps to the type (interface, class, TypedDict) of a variable.
           E.g. put cursor on a variable "user" and this takes you to the
           User type/interface that defines its shape.

  ;gr    Show all references
           Opens a Telescope picker listing every place this symbol is used
           in the project. Press Enter to jump to a reference.

  ;h     Hover docs
           Shows the type signature, docstring, and parameter info for the
           symbol under the cursor in a floating window. Press ;h again or
           q to close the float.

  ;rn    Rename symbol
           Renames the symbol under the cursor everywhere in the project
           (all references update together). Safe rename — not just a grep.

  ;ca    Code actions
           Opens a menu of quick-fixes and suggestions from the LSP:
           - Import a missing module
           - Fix a lint error automatically
           - Extract selected code to a function
           - Add missing type annotation
           The available actions depend on what the language server offers.

  ;ih    Toggle inlay hints
           Inlay hints are the small greyed-in bits of text the editor draws
           BETWEEN your code (not part of the file) showing inferred types,
           function return types, and parameter names at call sites — the same
           hints VS Code shows. They're off by default because they add visual
           noise; ;ih flips them on/off for the current buffer. Needs a language
           server that produces them (pyright for .py, ts_ls for .ts/.tsx).

  ;fo    Symbols in the current file · ;fw  symbols across the project
           Jump to a function/class/method by name. See |jwf-search|.

FORMAT ON SAVE ~

  Saving a .py/.ts/.tsx/.js file auto-formats it (ruff for Python, ts_ls for
  TS/JS). On Python it also sorts imports (ruff's isort). On JS/TS it applies
  eslint's auto-fixes. To save WITHOUT any of that:

  :noa w   `noautocmd write` — writes the buffer without firing any autocmds,
           so format-on-save and eslint's fix-on-save are both skipped. Works
           for any filetype. Add ! (`:noa w!`) to force-write a readonly file.

4. DIAGNOSTICS                                                 *jwf-diagnostics*

Diagnostics are errors, warnings, and hints reported by the LSP and ESLint.
They show as virtual text at the end of the line and markers in the sign column.

  ]d     Jump to the next diagnostic in the current file
  [d     Jump to the previous diagnostic in the current file

  ;fd    Open all project-wide diagnostics in a Telescope picker
           Lets you see every error and warning across all files at once.
           Press Enter to jump to the file and line. This is a ONE-SHOT picker —
           it closes when you pick something.

To see the full message for a diagnostic under the cursor:
  :lua vim.diagnostic.open_float()    (or just hover over it — it auto-shows)

TROUBLE — the persistent "Problems" panel ~

  trouble.nvim is VS Code's Problems panel: a list you keep OPEN on the side
  while you work through issues, grouped by file, updating live as you edit.
  Where ;fd is a quick pop-in-pop-out picker, Trouble stays put.

  ;xx    Toggle a panel of every diagnostic in the project
  ;xX    Toggle a panel of diagnostics for the CURRENT file only
  ;xs    Toggle a symbol outline of the current file (a live table of contents —
           functions/classes/methods; Enter jumps to one). Doesn't steal focus.
  ;xr    Toggle a panel of LSP references/definitions for the symbol under the
           cursor, opened on the right. Doesn't steal focus.

  Inside a Trouble panel: Enter opens the item, q closes the panel, and the
  same ;x* key toggles it back off.

5. GIT — CHANGED FILES & HISTORY                                      *jwf-git*

These commands help you understand what has changed and when.

SEEING WHAT FILES HAVE CHANGED:
  ;gs    Git status — Telescope picker of all changed/staged/untracked files
           Shows the same information as "git status" but lets you open any
           file directly. The letter on the left (M/A/D/?) is the git status.

  ;G     Open the fugitive status window (:Git)
           A full interactive git status panel. From here you can:
           - Press s on a file to stage it
           - Press u on a file to unstage it
           - Press = to toggle an inline diff for a file
           - Press cc to open a commit message buffer and commit
           - Press P to push
           - Press p to pull

SEEING HISTORY:
  ;gc    Git repo history — all commits, newest first (Telescope)
           Press Enter on a commit to see its full diff.

  ;gF    Git file history — commits that touched the current file only
           Great for answering "when was this function last changed and why?"
           Press Enter on a commit to see what that commit changed in the file.

  ;gl    Git log (one line per commit) via fugitive in the command window
           Quick overview without leaving the editor.

  ;gB    Git branches — switch branches via Telescope
           Press Enter to check out the selected branch.

GITHUB PERMALINK:
  ;gy    (normal mode)  Copy a permalink for the current line to clipboard
  ;gy    (visual mode)  Copy a permalink for the selected line range
           The URL is pinned to the current commit SHA, not the branch name,
           so it will always point to the exact same line of code even after
           the branch moves. Paste into Slack, GitHub comments, Jira tickets.

6. GIT — HUNK ACTIONS                                               *jwf-hunks*

A "hunk" is a contiguous block of lines that differ from the last commit.
The sign column (the narrow strip on the left) shows + for added lines,
~ for changed lines, and - where lines were deleted. These are set by gitsigns.

NAVIGATING HUNKS:
  ]h    Jump to the next changed hunk in the file
  [h    Jump to the previous changed hunk

REVIEWING CHANGES:
  ;gp    Preview hunk — shows a floating before/after diff for the current hunk
           Close it with q or Esc.

  ;gv    View full file diff split against HEAD
           Opens a vertical split: left side is the committed version,
           right side is your working copy.

  ;gb    Toggle inline blame
           Shows "abc1234  Jane Doe  3 days ago: fix button padding" at the
           end of the current line. Toggle it off when you are done.

STAGING & DISCARDING:
  ;ga    Stage hunk — adds just this block to the git index
           Lets you make a focused commit without staging unrelated changes
           in the same file.

  ;gu    Unstage hunk — removes it from the index (keeps your edits)

  ;gx    Reset hunk — discards local changes in this block, reverts to HEAD
           WARNING: this is destructive. The changes are gone.

7. TERMINAL / SHELL                                               *jwf-terminal*

toggleterm gives you persistent floating terminals that keep their shell
session alive across toggles. Each terminal has a numeric id; the SAME id
always reopens the SAME shell session, so you can keep several long-running
processes going at once and flip between them without losing state.

OPENING SPECIFIC SHELLS:
  ;sh    Toggle shell 1 (floating)
           The shell session persists — if you start a dev server here,
           it keeps running while you edit code and is still there when
           you toggle the terminal back open.

  ;sv    Toggle shell 1 as a horizontal split instead of a float
           Splits the window horizontally with the terminal at the bottom.
           Useful for running a watcher (npm run dev, pytest --watch) while
           keeping the code visible above. Same id as ;sh, so it flips the
           one session between float and split rather than making a new one.

  ;s2    Toggle shell 2 (floating)
  ;s3    Toggle shell 3 (floating)
  ;s4    Toggle shell 4 (floating)
           Each is a completely separate shell session. Run a dev server in
           shell 1, a test watcher in shell 2, a Python/Node REPL in shell 3,
           and so on — every one keeps running while hidden.

  ;s2v   Toggle shell 2 as a horizontal split (like ;sv is for shell 1)
  ;s3v   Toggle shell 3 as a horizontal split
  ;s4v   Toggle shell 4 as a horizontal split
           Same id as the matching ;s2–;s4 above, so the float and split
           shortcuts flip the one session between layouts rather than opening
           a new shell.

OPENING ANY NUMBERED SHELL ON THE FLY:
  Ctrl-\      Toggle shell 1 from anywhere (normal or insert mode).
  2 Ctrl-\    Prefix the toggle with a COUNT to open that numbered shell.
  3 Ctrl-\    e.g. 5<C-\> opens (or hides) shell 5 — no keymap needed for it.
           This is how you go beyond the named ;s2–;s4 shortcuts: any id you
           type a count for becomes its own persistent session.

MANAGING ALL OPEN TERMINALS:
  ;ss    :TermSelect — opens a menu listing every terminal that is currently
           open, by id and name. Pick one to jump straight to it. Best way to
           find a shell when you have several and don't remember which id it
           got.

  ;sq    :ToggleTermToggleAll — show or hide ALL terminals at once. Handy to
           clear the screen of every floating shell, then bring them all back
           with the same key. The sessions keep running while hidden.

Inside a terminal buffer:
  Esc Esc   Return to normal mode (does NOT close the terminal or kill
             the process — the shell keeps running). A single Esc is passed
             straight through to the running program, so TUIs that depend on
             Esc — Claude Code, vim, fzf, less — receive it normally.
  i         Enter insert/typing mode again
  Ctrl-\    Toggle the terminal closed
  :q        Actually close and kill the terminal session

8. WINDOW NAVIGATION                                               *jwf-windows*

Neovim splits the screen into "windows" (panes). Each window shows a buffer.

  Ctrl-w h    Move focus to the window on the LEFT
  Ctrl-w l    Move focus to the window on the RIGHT
  Ctrl-w j    Move focus to the window BELOW
  Ctrl-w k    Move focus to the window ABOVE
  Ctrl-w w    Cycle through all open windows

Resizing windows:
  Ctrl-w +    Increase height
  Ctrl-w -    Decrease height
  Ctrl-w >    Increase width
  Ctrl-w <    Decrease width
  Ctrl-w =    Equalise all window sizes

Opening splits:
  :sp         Horizontal split (same file)
  :vsp        Vertical split (same file)
  :sp <file>  Horizontal split opening a different file

TABS                                                                  *jwf-tabs*

There are TWO different "tab" things in this config. Know which you mean:

  BUFFER TABS  — the bar across the top showing one tab per open FILE, exactly
                 like VS Code. Provided by the bufferline plugin. This is what
                 you usually want.
  NATIVE TABS  — Vim's :tabpages, which are whole WINDOW LAYOUTS (sets of
                 splits), not files. Useful for keeping two different split
                 arrangements around. Rarely needed day to day.

────────────────────────────────────────────────────────────────────────────
BUFFER TABS (VS Code style — the top bar)                        *jwf-buffertabs*

Every file you open shows as a numbered tab along the top. The number on the
tab matches the ;<number> jump keys below. Diagnostics (error/warn counts)
show on each tab. Closing a tab just closes that file; it does not touch your
window layout.

  Shift-l            Go to the NEXT tab            (BufferLineCycleNext)
  Shift-h            Go to the PREVIOUS tab        (BufferLineCyclePrev)
  ;1 ;2 … ;9         Jump straight to tab number N
  ;bp                Pick a tab by letter — labels appear on each tab, press
                     the letter to jump there (fastest with many tabs open)
  ;bd                Close the current tab (the buffer). Switches to the
                     PREVIOUS tab first, so the window keeps showing code and
                     the file tree stays put instead of expanding fullscreen.
  ;bo                Close every OTHER tab, keep the current one
  ;bl                Move the current tab one place to the RIGHT (reorder)
  ;bh                Move the current tab one place to the LEFT
  ;fb                Fuzzy-pick an open file by name — scales better than the
                     bar once you have a lot of files open (|jwf-quick|)
  Ctrl-^             Toggle between the current and previous file (very fast)

You can also left-click a tab to switch, or click its x to close it.

────────────────────────────────────────────────────────────────────────────
NATIVE TABS (window layouts)                                     *jwf-nativetabs*

A native tab is a whole WINDOW LAYOUT, not a single file. Buffers are shared —
every open file is reachable from every native tab. Use these only when you
want a genuinely different split arrangement for a different task (e.g. one tab
with code+test side by side, another showing a single full-screen file).

  :tabnew            Open a new empty native tab
  :tabnew <file>     Open <file> in a new native tab
  :tab split         Put the CURRENT file in a new native tab (keeps it open)
  Ctrl-w T           Move the current split OUT into its own new native tab
  gt                 Next native tab            :tabn   (same)
  gT                 Previous native tab        :tabp   (same)
  1gt 2gt 3gt        Jump to native tab by number (count + gt)
  :tabs              List native tabs and the windows/buffers in each
  :tabclose  :tabc   Close the current native tab (its buffers stay open)
  :tabonly   :tabo   Close every OTHER native tab
  :tabmove 0         Move this native tab to the front
  :tabmove +1 / -1   Shift it right / left one spot

Heads-up: when native tabs are open, the top bar shows the NATIVE tabs instead
of the bufferline buffer tabs. That's expected — leave native tabs alone and
the buffer-tab bar comes back.

9. DEBUGGING (DAP) — Django / debugpy                                *jwf-debug*

Neovim uses the Debug Adapter Protocol (DAP) — the same protocol VS Code uses.
nvim-dap acts as the DAP client; debugpy (running inside the Docker container)
acts as the DAP server. The Django dev server runs TWO uwsgi workers, each with
its own debugpy listener, on ports 5678 and 5680.

FIRST TIME? — THE 30-SECOND VERSION ~

  Never debugged in Neovim? It's the same idea as VS Code's debugger, just
  driven by keys instead of clicks. The whole loop is four keystrokes:

    1. Put your cursor on a line you want to freeze on, press  ;db
       (a red ● appears — that's a "breakpoint").
    2. Press  ;da  to attach. A panel opens on the left (variables, call
       stack) and one on the bottom (a Python console). Debugger is now live.
    3. Load the page / hit the API endpoint that runs that line in your
       browser or with curl. The moment Django reaches your ● it FREEZES,
       and a green ▶ shows the exact line it stopped on.
    4. Now poke around while it's frozen:
         ;dh  on a variable  → shows its value in a popup
         ;dr                 → opens a Python prompt IN that frozen moment,
                               so you can type  request.user  and see the answer
         ;dn / ;di / ;do     → run the next line / step into a call / step out
         ;dc                 → un-freeze, run until the next ● (or the next
                               request hits it again)
       Done? Press  ;dt  to detach. Your server keeps running the whole time.

  Mental model: a breakpoint is a "pause here" flag. Attaching connects Neovim
  to the already-running Django so it can honor those flags. Nothing you do
  here changes or restarts the server — you're just watching it run.

  Gotchas worth knowing up front:
    - Nothing happens until a real request actually runs the line. Set the
      breakpoint, THEN trigger the code — a breakpoint on code that never runs
      just never fires.
    - Always attach with ;da (both workers), not ;dc (one worker). Django
      spreads requests across two workers; attach to one and roughly half your
      requests slip past the breakpoint and it looks "broken."
    - If the ● shows hollow ○ instead of solid, debugpy couldn't map the file
      (see PATH MAPPINGS below) — the breakpoint won't fire.
    - Errors do NOT auto-pause the debugger by default — only your ;db lines
      do. Press ;de to toggle stopping on unhandled exceptions too (see
      EXCEPTION BREAKPOINTS below).

  The step-by-step below is the same thing, in more detail.

STEP 0 — Nothing to start
  The Codespaces web container sets ENABLE_DEBUGPY=true, so debugpy is always
  listening on 5678 + 5680 whenever your normal dev server is running. There is
  no separate debug container and no bin/debugpy.sh — just have the app running.

STEP 1 — Set breakpoints
  Open any Django Python file and press  ;db  on the line you want to pause on.
  A red dot ● appears in the sign column. You can set as many as you like.

STEP 2 — Attach
  Press  ;da  (attach to Django).
  This attaches to BOTH uwsgi workers (5678 + 5680) at once. This matters:
  uwsgi load-balances each web request across the two workers, so if you attach
  to only one, every request the other worker serves sails past your breakpoint.
  ;da is the nvim equivalent of VS Code's "Django Debug All Workers" compound.

  The DAP UI opens automatically showing:
    - Left panel: Variables, Watches, Call Stack, Breakpoints list
    - Bottom panel: REPL + Django console output
  Django keeps running normally and waiting for requests.

  (;dc instead attaches to a single worker you pick — rarely what you want.)

STEP 3 — Trigger the code
  Make a web request that exercises the code path with your breakpoint.
  Django will pause when it hits the breakpoint and the DAP UI highlights
  the current line with a green arrow ▶.

STEP 4 — Inspect and step
  ;dn    Step over — run this line, stop at the next
  ;di    Step into — follow a function call inside it
  ;do    Step out — finish the current function, stop at the caller
  ;dc    Continue — run until the next breakpoint

  ;dh    Hover — show the value of the variable under the cursor
  ;dr    REPL — open an interactive Python console in the current frame
           Type any Python expression to evaluate it in the paused scope,
           e.g.:  user.email  or  self.request.POST

STEP 5 — Terminate
  ;dt    Disconnect from debugpy. Django continues running normally.
         Both worker sessions detach; the dev server is unaffected.

ALL DEBUG SHORTCUTS:
  ;da    Attach to Django — both uwsgi workers (5678 + 5680) at once
  ;dc    Continue / attach a single worker
  ;db    Toggle breakpoint on current line
  ;dB    Conditional breakpoint (prompts for a Python expression)
  ;dn    Step over
  ;di    Step into
  ;do    Step out
  ;dt    Terminate debug session
  ;dr    Open interactive REPL
  ;dh    Hover: show value of expression under cursor
  ;dl    List all breakpoints
  ;dC    Clear ALL breakpoints
  ;du    Toggle DAP UI open/closed manually
  ;de    Toggle stopping on unhandled exceptions on/off (see below)

EXCEPTION BREAKPOINTS (stopping on errors, not just ;db lines):
  Besides the ;db line breakpoints, debugpy can also pause the moment an
  exception occurs. By default this is turned OFF here — only your ;db lines
  stop — so a stack trace doesn't yank you into the debugger unless you asked.

  Press  ;de  to toggle it. ON pauses whenever an exception goes unhandled
  (debugpy's "uncaught" + "userUnhandled" filters — i.e. it escapes your code
  or crashes the request); press ;de again to turn it back OFF. The toggle
  applies to both worker sessions at once, and a message confirms the new
  state. It also resets to OFF when you re-attach with ;da.

  ;de deliberately leaves out the noisy "raised" filter (which stops on every
  raise, even ones caught by a try/except). If you're hunting where a swallowed
  error starts and want that, set it manually with:
    :lua require("dap").set_exception_breakpoints({ "raised" })

PATH MAPPINGS (why breakpoints work):
  The source code exists in two places:
    Host (Neovim):      /workspaces/web/
    Container (Django): /web/
  debug.lua maps these paths so breakpoints set in Neovim match the file
  paths that debugpy reports. Without this, breakpoints silently never fire.
  A second mapping (/workspaces/web/venv/lib/python3.11/site-packages →
  /usr/local/lib/python3.11/site-packages) lets you step into installed
  packages. These mirror .vscode/launch.json's Django configs.

11. WORKFLOW RECIPES                                               *jwf-recipes*

Task-oriented walkthroughs that chain the individual commands above. Each
recipe lists the keys in order with what happens at each step.

────────────────────────────────────────────────────────────────────────────
RECIPE A — Rename a symbol everywhere it is used                *jwf-recipe-rename*

  1. Put the cursor on the function/variable/class name.
  2. ;rn          Type the new name, press Enter.
                  The LSP rewrites every reference across the whole project
                  in one shot — not a text search, so it won't touch a
                  same-named symbol that is actually something else.
  3. :wa          Save all changed buffers. format-on-save runs for each.

  Want to eyeball the usages before renaming? Press ;gr first to list every
  reference in a Telescope picker, then ;rn when satisfied.

────────────────────────────────────────────────────────────────────────────
RECIPE B — Find every usage of something, then fix each one     *jwf-recipe-usages*

  1. Cursor on the symbol  →  ;gr      Lists all references (Telescope).
  2. Ctrl-q                            Send the whole list to the quickfix.
  3. :cn / :cp                         Walk forward/back through each usage,
                                       editing as you go.
  4. :copen                            Open the quickfix window to see the
                                       full list; Enter on a row jumps to it.

  Don't have an LSP symbol (e.g. a string key, a CSS class)? Use ;ff to grep
  the literal text instead, then Ctrl-q from that picker.

────────────────────────────────────────────────────────────────────────────
RECIPE C — Stage a focused, partial commit                      *jwf-recipe-partial*

  You changed three things in one file but want to commit only one of them.

  1. ]h / [h       Jump between the changed hunks in the file.
  2. ;gp           Preview the hunk under the cursor to confirm it's the one.
  3. ;ga           Stage just that hunk (leaves the others unstaged).
  4. Repeat 1–3 for any other hunks that belong in this commit.
  5. ;G            Open fugitive. Confirm only the intended hunks are staged.
  6. cc            Write the commit message, :wq to commit.

  Staged the wrong hunk?  ;gu  unstages it without losing the edit.

────────────────────────────────────────────────────────────────────────────
RECIPE D — Debug a single Django request end to end             *jwf-recipe-debug*

  1. Have your normal dev server running (debugpy is always listening in the
     Codespaces web container — nothing extra to start).
  2. Open the view file, ;db on the line to pause at.
  3. ;da                               Attach to both workers. DAP UI opens.
  4. Trigger the request (browser / curl). Execution pauses at the line.
  5. ;dh   over a variable             Show its value inline.
     ;dr                               REPL — evaluate  self.request.POST  etc.
     ;dn / ;di / ;do                   Step over / into / out.
  6. ;dc to run to the next breakpoint, or ;dt to detach when done.

  Full reference: |jwf-debug|.

────────────────────────────────────────────────────────────────────────────
RECIPE E — Project-wide search and replace                      *jwf-recipe-replace*

  1. ;sr                               Open spectre.
  2. Type the search term (top field) and the replacement (second field).
                                       Regex supported.
  3. Review the match list — each shows the file and the before/after.
     Put the cursor on a match and press dd to exclude it from the replace.
  4. <leader>R  (that is  ;R )         Execute the replace across all files.
  5. :wa                               Save everything.

  For a quick single-symbol rename in code, prefer ;rn (Recipe A) — it is
  type-aware. Use spectre for text that the LSP doesn't track (strings,
  comments, config values, filenames in content).

────────────────────────────────────────────────────────────────────────────
RECIPE F — Share a permalink to code in Slack / a PR            *jwf-recipe-link*

  1. Put the cursor on the line  (or select a range in visual mode).
  2. ;gy                               Copies a github.com permalink to the
                                       system clipboard, pinned to the current
                                       commit SHA so it never drifts.
  3. Cmd-v into Slack / GitHub / Jira.

  The link points at the committed code. If your line only exists in unpushed
  local work, push first or the URL will 404 for others.

────────────────────────────────────────────────────────────────────────────
RECIPE G — Run a dev server and tests side by side              *jwf-recipe-procs*

  1. ;sh                               Shell 1 (floating). Start the dev server
                                       here, e.g.  npm run dev.
  2. Esc Esc                           Back to normal mode (server keeps
                                       running in the background).
  3. ;s2                               Shell 2 — a separate session. Run the
                                       test watcher here, e.g. pytest --watch.
  4. ;sh / ;s2                         Toggle either shell open or closed; both
                                       processes keep running the whole time.

  Prefer them on screen at once? ;sv opens a horizontal split terminal under
  the code instead of a floating overlay.

────────────────────────────────────────────────────────────────────────────
RECIPE H — Find out when and why a line changed                 *jwf-recipe-blame*

  1. ;gb                               Toggle inline blame — shows the commit,
                                       author, and date at the end of the line.
  2. ;gF                               File history (Telescope). Enter on a
                                       commit shows exactly what it changed
                                       in this file.
  3. ;gc                               Whole-repo history if you need wider
                                       context around that commit.

────────────────────────────────────────────────────────────────────────────
RECIPE I — Clean up errors and lint in a file                   *jwf-recipe-clean*

  1. ]d / [d                           Walk through each diagnostic in the file.
  2. ;ca   on a flagged line           Code actions — auto-import, apply the
                                       suggested fix, add a type annotation.
  3. :w                                Save. On .py/.ts/.tsx/.js this runs
                                       format-on-save; on JS/TS, ESLint also
                                       auto-fixes everything it can.
  4. ;fd                               Project-wide diagnostics picker to confirm
                                       nothing is left across the whole project.

12. TMUX COOKBOOK                                                     *jwf-tmux*

tmux runs shell sessions inside a server that OUTLIVES your terminal. Close the
terminal, lose the ssh connection, reboot your laptop's terminal app — the
session and everything running in it keep going. You "attach" to see it, and
"detach" to leave it running. All of this is separate from Neovim's toggleterm
(|jwf-terminal|); tmux is at the shell level and survives across nvim restarts.

VOCAB
  server     One background process holding ALL your sessions. Starts on first
             use, dies when the last session is killed.
  session    A named workspace (like a project). Holds windows.
  window     A tab inside a session. Holds panes.
  pane       A split within a window (one shell each).
  prefix     The key you press before any tmux command WHILE ATTACHED.
             Default is  Ctrl-b  (written  C-b  below). Press it, release,
             then press the command key.

────────────────────────────────────────────────────────────────────────────
CREATING SESSIONS                                                *jwf-tmux-create*

  tmux                         Start a session (auto-named 0, 1, 2…) and attach.
  tmux new -s work             Start a session NAMED "work" and attach to it.
  tmux new -d -s work          Create "work" DETACHED — it runs in the
                               background, your terminal stays where it is.
                               (This is what the cs_claude alias does.)
  tmux new -d -s work 'CMD'    Create detached AND run CMD in it, e.g.
                               tmux new -d -s dev 'npm run dev'
                               The session lives as long as CMD runs (or longer
                               if CMD is a shell).

────────────────────────────────────────────────────────────────────────────
LISTING & ATTACHING                                             *jwf-tmux-attach*

  tmux ls                      List all sessions (name, #windows, attached?).
  tmux attach -t work          Attach to "work".  (tmux a -t work  for short)
  tmux attach                  Attach to the most recent session.
  tmux attach -t work -r       Attach READ-ONLY — watch without being able to
                               type. Good for tailing a live session safely.
  C-b d                        DETACH from the current session (while attached).
                               Everything keeps running; you drop back to your
                               normal shell.
  tmux switch -t other         Jump straight to another session (when already
                               attached to one).

────────────────────────────────────────────────────────────────────────────
TAILING OUTPUT WITHOUT ATTACHING                                  *jwf-tmux-tail*

Yes — several ways, no attach needed:

  Snapshot (what's on screen right now):
    tmux capture-pane -t work -p
        -p prints to stdout. Dump it, grep it, redirect it:
        tmux capture-pane -t work -p | tail -30
        tmux capture-pane -t work -p > /tmp/out.txt

  Snapshot INCLUDING scrollback (not just the visible screen):
    tmux capture-pane -t work -p -S -200      last 200 lines of history
    tmux capture-pane -t work -p -S -          the ENTIRE scrollback

  Target a specific window/pane with  session:window.pane :
    tmux capture-pane -t work:0.1 -p

  LIVE follow (true tail -f, updates as output arrives):
    tmux pipe-pane -t work -o 'cat >> /tmp/work.log'   start mirroring
    tail -f /tmp/work.log                              follow it anywhere
    tmux pipe-pane -t work                             stop mirroring (no arg)

  Send a command into a session without attaching:
    tmux send-keys -t work 'ls -la' Enter
        Enter is a KEY NAME here, so it actually runs the command. Great for
        driving a detached session from a script.

────────────────────────────────────────────────────────────────────────────
DELETING / KILLING SESSIONS                                       *jwf-tmux-kill*

  tmux kill-session -t work    Kill ONE session (and everything in it).
  tmux kill-server             Kill EVERYTHING — all sessions at once.
  C-b &                        Kill the current WINDOW (asks to confirm).
  C-b x                        Kill the current PANE (asks to confirm).
  exit  (or C-d) in the pane   Closes that pane; last pane closes the window;
                               last window ends the session normally.

────────────────────────────────────────────────────────────────────────────
WHILE ATTACHED — the everyday keys                                *jwf-tmux-keys*

All of these are pressed AFTER the prefix  C-b  (press C-b, let go, then the key):

  Windows (tabs):
    C-b c        Create a new window
    C-b ,        Rename the current window
    C-b n / p    Next / previous window
    C-b 0..9     Jump to window by number
    C-b w        Visual list of windows to pick from

  Panes (splits):
    C-b %        Split vertically (side by side)
    C-b "        Split horizontally (top/bottom)
    C-b ← ↑ ↓ →  Move focus between panes (arrow keys)
    C-b o        Cycle to the next pane
    C-b z        Zoom the current pane fullscreen (toggle)
    C-b {  }     Swap the pane with the previous / next one
    C-b space    Cycle through pane layouts

  Sessions:
    C-b s        Visual list of sessions to switch between
    C-b $        Rename the current session
    C-b d        Detach (leave it all running)

  Copy / scroll:
    C-b [        Enter copy mode — now scroll with arrows / PageUp / C-u / C-d,
                 search with /. Press q to leave.
                 (In copy mode: Space starts a selection, Enter copies it.)

────────────────────────────────────────────────────────────────────────────
RECIPE — long-running job you can walk away from             *jwf-tmux-recipe-job*

  1. tmux new -d -s build 'make && ./deploy.sh'    Start it detached.
  2. tmux capture-pane -t build -p | tail -20      Peek at progress anytime.
     (or)  tmux attach -t build                    Watch it live; C-b d to leave.
  3. tmux ls                                        Still there? "build" listed =
                                                    still running.
  4. tmux kill-session -t build                     Clean up when done.

  Closing your terminal / losing ssh at any point does NOT stop the job — that
  is the whole point of running it under tmux.

13. AI / CLAUDE CODE WORKFLOW                                           *jwf-ai*

Two things make Neovim a real pair-editor with Claude Code:
  - claudecode.nvim — runs a WebSocket server inside Neovim that the `claude`
    process auto-connects to (same machine, so it works over SSH / in a
    Codespace with no port forwarding). Claude can then read your selection,
    the files you add, and hand edits back as diffs you accept in Neovim.
  - autoread — when Claude writes files on disk, your open buffers refresh
    automatically instead of going stale (see below).

All commands sit under the ;a ("AI") prefix.

STARTING / FOCUSING CLAUDE                                          *jwf-ai-start*
  ;ac    Toggle the Claude Code terminal in a split on the right. The claude
         it launches auto-connects to Neovim, so selections and @-mentions
         from here land in that session.
  ;af    Focus the Claude terminal (jump the cursor into it).
  ;ar    Start Claude with --resume (pick a past session to reopen).
  ;aC    Start Claude with --continue (reopen the most recent session).

  Note: use ;ac to launch claude so the editor integration connects. A claude
  you start by hand in a ;sh toggleterm shell will run fine but won't be linked
  to Neovim (no selection/diff sharing).

SENDING CONTEXT TO CLAUDE                                             *jwf-ai-send*
  ;as    (visual mode) Send the highlighted lines to Claude as context —
         the fastest way to say "look at THIS".
  ;ab    Add the current file to Claude's context (whole buffer).
  ;as    (in the file tree) Add the file/folder under the cursor to context.

REVIEWING CLAUDE'S EDITS                                             *jwf-ai-diff*
  When Claude proposes a change it opens as a diff IN Neovim:
  ;aa    Accept the diff (apply the change).
  ;ax    Reject the diff (discard it).

  ;gV    Open diffview: a full-window review of EVERY file changed in your
         working tree — ideal after Claude edits several files at once.
  ;gQ    Close diffview.
  ;gh    Diffview history of just the current file.

PASTE-INTO-ANY-CHAT HELPERS                                          *jwf-ai-yank*
  These don't need the integration — they just fill your clipboard so you can
  paste into a web chat, a PR comment, Slack, etc. Both also work over SSH
  (they write the + register, which goes to your Mac clipboard via OSC 52).

  ;ay    (visual mode) Yank the selection WITH a "path:startline-endline"
         header prepended, so the snippet says where it came from.
  ;ae    Yank every diagnostic (error/warning) in the current file, each as
         "Lnn [SEVERITY] message" — paste the whole list instead of a
         screenshot when asking the model to fix errors.

AUTO-RELOAD OF EXTERNALLY EDITED FILES                             *jwf-ai-reload*
  Claude Code (and git checkouts, formatters, rebases) change files on disk
  outside Neovim. With autoread on plus a checktime autocmd, buffers you have
  open re-read themselves when you refocus the window, switch to the buffer,
  or leave a terminal — so you never edit a stale copy or overwrite Claude's
  work on the next save. Nothing to press; it just happens.
  If you ever want to force it manually:  :checktime
